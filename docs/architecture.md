# Architecture

Skyvern automates browsers using a Vision LLM loop rather than pre-written selectors. This page explains why that matters, how the pieces fit together, and what happens at each layer when you run a task.

## Why Vision LLM instead of selectors

Traditional browser automation works by locating DOM elements with XPath or CSS selectors (`#submit-button`, `.checkout-form input[type="text"]`). This is fast and reliable — until the website changes. A single redesign can silently break dozens of automations, and sites that render content dynamically or behind anti-bot measures are nearly impossible to scrape reliably this way.

Skyvern's approach is different: instead of teaching the system where to click, you describe *what you want done*. At each step, Skyvern takes a screenshot of the current browser state and asks a Vision LLM to decide what action to take next. The LLM reads the screenshot visually, the same way a human would, and returns an action. Playwright then executes that action, another screenshot is taken, and the loop continues.

The tradeoff is speed: each step involves an LLM call, which takes 1-5 seconds. For automations that would otherwise require manual maintenance or break constantly, that cost is worth it.

## The execution loop

Here is the core loop for a single task:

```
1. Task received (goal + URL)
         ↓
2. Browser opens the target URL
         ↓
3. Playwright takes a screenshot
         ↓
4. Vision LLM analyzes screenshot + action history
         ↓
5. LLM returns an action (click, input_text, select_option, ...)
         ↓
6. Playwright executes the action
         ↓
7. Step result and screenshot saved as artifacts
         ↓
8. Loop back to step 3 — until COMPLETE or TERMINATE is returned
```

When the LLM determines the goal has been achieved, it returns a `complete` action, which ends the loop. If the task is impossible (wrong URL, insufficient permissions, etc.), it returns `terminate` instead. Each iteration is a **Step**, and steps are saved to the database so you can inspect every decision the agent made.

## Layered architecture

The codebase is organized into four layers:

```
User Interfaces (Web UI, Python SDK, TypeScript SDK)
            ↓
FastAPI Application  (skyvern/forge/api_app.py)
  ├─ CORS, Auth, Request logging middleware
  ├─ base_router  ─── /api/v1/* endpoints
  └─ legacy routers for backward compatibility
            ↓
Routes Layer  (skyvern/forge/sdk/routes/)
  ├─ agent_protocol.py  — tasks, workflows, runs
  ├─ scripts.py         — script CRUD and execution
  ├─ credentials.py     — secrets management
  ├─ browser_sessions.py — browser lifecycle
  └─ streaming/          — WebSocket and SSE handlers
            ↓
Services Layer  (skyvern/services/)
  ├─ task_v2_service.py  — task orchestration
  ├─ script_service.py   — script generation and execution
  └─ workflow_service.py — workflow execution
            ↓
Core Layer  (skyvern/webeye/, skyvern/forge/agent.py)
  ├─ BrowserManager   — Playwright browser lifecycle
  ├─ Agent            — LLM agent loop (execute_step)
  ├─ Action handlers  — translates LLM output to Playwright calls
  └─ LLM APIs         — multi-provider via LiteLLM
            ↓
Data Layer  (SQLAlchemy + PostgreSQL, S3 / Azure Blob)
```

Each layer only talks to the layer directly below it. Route handlers call service functions; service functions call agent and browser code; nothing in routes directly touches the database or browser.

## Key abstractions

### Tasks

A **Task** is a single unit of work: a URL, a goal, and optionally a data extraction schema. Tasks go through a lifecycle: `created → queued → running → completed` (or `failed`, `timed_out`, `terminated`). Each step in the loop is stored as a **Step**, and each step can produce **Artifacts** (screenshots, LLM responses, recordings).

Tasks are defined in `forge/sdk/db/models.py:TaskModel` and executed by `forge/agent.py:Agent.execute_step()`.

### Workflows

A **Workflow** is a directed acyclic graph of **Blocks**. Blocks are the building blocks of multi-step automations. Each block has a type (task, script, for_loop, conditional, code, text_prompt, http_request, send_email, and more), a label used for output references, and parameters. Blocks are defined in YAML and stored in the database.

When a workflow runs, a **WorkflowRun** is created to track the execution. Block outputs are stored as `{label}_output` and can be referenced in downstream blocks using Jinja2 templating: `{{ my_task_output }}`.

### Scripts

A **Script** is executable Python or TypeScript automation code. Scripts can be generated by the LLM from a goal description, reviewed, deployed, and reused. They run inside a sandboxed environment with access to a `SkyvernPage` object that exposes AI-augmented Playwright commands.

### Browser sessions

By default, each task gets a fresh browser context that is destroyed when the task completes. **Persistent browser sessions** let you save and restore a browser's state (cookies, local storage, login session) across tasks. Session data is stored in S3 or Azure Blob.

### Credentials

Credentials (passwords, TOTP secrets, credit card details) are stored encrypted in the database. They can also sync from external vaults (Bitwarden, 1Password, Azure Key Vault). Credentials are never returned in API responses — only non-sensitive metadata like the username and last-four digits.

### Artifacts

Every screenshot, LLM response, page recording, and extracted data file is stored as an **Artifact**. Artifacts are saved to S3 or Azure Blob Storage and exposed via signed URLs with 24-hour expiration. This gives you a full audit trail of every task execution.

## LLM providers

Skyvern uses [LiteLLM](https://github.com/BerriAI/litellm) as a unified interface over multiple LLM providers. The active provider is set with `LLM_KEY` in your `.env` file. Supported providers include OpenAI, Anthropic, Google Gemini, Azure OpenAI, Amazon Bedrock, Ollama, and several others. Vision support (required for screenshot analysis) depends on the model — `LLM_CONFIG_SUPPORT_VISION` must be `true`.

## Multi-tenancy

Every resource (task, workflow, credential, artifact) belongs to an **Organization**. Authentication is done with an API key that maps to an organization. All members of an organization share full access to that organization's resources. JWT tokens are used for the web UI session; API keys are used for SDK and direct API calls.

## Common misconceptions

**"Skyvern takes a screenshot once and figures everything out."** No — it takes a screenshot at every step. The LLM only sees the current state of the page, not past states (though the action history is included in the prompt).

**"Skyvern replaces Playwright."** Skyvern sits on top of Playwright, using it to execute actions. You can still use plain Playwright selectors in scripts if you know the site won't change.

**"Skyvern works with any LLM."** It works with any LLM that supports vision (image input). Text-only models won't work for the core automation loop, though they can be used for sub-tasks like code generation and prompting.
